# 省電力・発熱 調査報告（2026-07-05）

- 対象: `proto3d`（three r169 / WebGL2 / EffectComposer）。症状=iPhone 17 Pro でゲーム内1日（実時間およそ10〜15分の連続描画）プレイすると発熱。
- 目的: **画質・品質を維持したまま**、電池消費と発熱を最大限削る。カクつきも増やさない。
- この文書は調査のみ。コードは未変更。実装は別セッション（Opus 4.8）が本書を読んで行う。
- 実測環境: 開発機のheadless Edge（swiftshader）。**コール数・オブジェクト数・行列コストは実機と同じ構図**。GPU時間の絶対値だけは実機と別物（最終確認はiPhoneのSafari Webインスペクタで）。

---

## 0. 結論（先に要点）

発熱の正体は単一の犯人ではなく、**「1フレームの仕事が全方位でやや重い × 30fpsで10〜15分回し続ける」の積分**。リークやスパイクの異常は無い（下記実測）。その上で、**見た目を1ピクセルも変えずに削れる無駄が大量に残っている**：

| # | 見た目不変の無駄 | 実測根拠 |
|---|---|---|
| 1 | 静的な3.1万ノードの行列を毎フレーム再計算 | updateMatrixWorld 15.9ms/回（全OFFなら3.7ms）×最大2回/フレーム＝**約24ms/フレームのCPUが無駄**（開発機実測） |
| 2 | 霧に完全に沈んだ距離のオブジェクトを描画 | 可視3176中**538ドロー（17%）**が fog.far+10m 以遠＝画面上は霧色のみ |
| 3 | ポストプロセスRTが16bit（HalfFloat） | three r152以降のEffectComposer既定。8bit化で**ポスト帯域が算術的に半減** |
| 4 | `preserveDrawingBuffer: true` | TBDR（Apple GPU）で毎フレームのバッファ保持コスト。撮影は同フレーム内キャプチャで代替可 |
| 5 | 無音のオシレータが常時発振 | 雨BGMパッド**17本**＋夕夜BGMパッド6本が、鳴っていない時間も稼働 |
| 6 | 完全静止中もプリパス（シーン二重描画）が1/2頻度で継続 | 静止が続けば0回でよい（変化時に即復帰） |

これらを合わせると、**同じ絵のまま CPU仕事を3〜5割・GPU帯域を3〜4割削れる見込み**。加えて「ほぼ不可視」「要ユーザー判断」の施策が下記C・D節にある。

- **InstancedMeshは可視4個のみ**。つまり町の大半は「1物体=1ドロー」。Safari/WebKitはWebGLコール単価が高い（Wonderland Engine報告）ため、**3千超のコール発行そのものがCPU電力**。
- 透過マテリアル406（重ね描きコスト・参考値）。
- 霧外の無駄: `scripts/_dcfog.mjs` 新設。fog.far(この時470m)+10m以遠に**538ドロー/0.05Mtri**。地形チャンクは距離カリング済みだが、**シーン直下の木・杭・建物小物は対象外**だった。

### 1-3. 毎フレームのCPU（棚卸し・行番号つき詳細は§6）

- **行列**: `matrixAutoUpdate=false` はコード全体で0箇所。31,538ノード全てが毎renderで行列合成。実測（開発機）: updateMatrixWorld 1回=15.9ms、全ノードOFF時=3.7ms。renderは最大2回/フレーム（プリパス+本描画）＝**約24〜32ms/フレームが静止物の行列計算**。
- **update(dt)**: 約40サブシステムが毎フレーム全走査。通行人/谷戸住人/祭りには計算LOD(60m)・描画カリング(130m)あり。**猫・犬・蝶・トンボ・カラス・ツバメ・雀・蛙・メダカは距離カリング無し**で、毎フレーム個体ごとに `heightAt`（DEMバイリニア補間）を実行＝1フレームで数百回のDEMサンプリング。
- アロケーションは概ねプール済み（優秀）。`pickRoadPt` の配列生成と近接UI判定の `Math.hypot` 群が残る程度。
- setInterval系の裏仕事は無し（UI表示中のみ3つ）。

### 1-4. GPUパイプライン（1フレーム・地上歩行・昼）

```
[プリパス] 法線/深度RT(フル解像度・far300絞り済) ← シーン全体をもう1回描画（移動中は毎フレーム/静止中1/2）
[本描画]  RenderPass(3.4k calls)
[ポスト]  godray(フル解像・昼のみ) → bloom(半解像・内部多段) → grade(フル解像・重フラグ)
          → dof(フル解像・歩行時) → ink(フル解像) ＝ フル解像級パス約5〜6本/フレーム
RT型: EffectComposer既定の HalfFloatType（16bit×4ch）＝8bit比で帯域2倍
その他: preserveDrawingBuffer:true（毎フレームの保持コスト）・DPR上限1.25・30fps上限
```

- 影は既に最適: 太陽1灯・手動更新（エリア切替 or tdayが0.012動いた時だけ）。
- 音: 常時ループ=環境MP3×4+風+雨+風鈴+川、**雨BGMパッド17oscと夕夜パッド6oscは無音時間帯も発振**。蝉合成はトリガ式で常時ノード0（優秀）。`latencyHint:'playback'` 済み。

### 1-5. Web最新事情（2025-26・出典は§7）

- iOS SafariのrAFは**既定60Hz上限**（ProMotion 120Hzでも）。ページ側からのフレームレート指定APIは無い＝自前間引き（現行30fps）が正解。**iOS低電力モード時は自動で30fps**＝本ゲームは見た目が変わらない。
- `powerPreference:'low-power'`は単一GPUのiPhoneで実効なし。`desynchronized`はSafari未対応。
- **WEBGL_multi_drawはSafari 15+で利用可**＝three.jsのBatchedMeshが使える。
- WebGPUはSafari 26で出荷済みだが、カスタムGLSL多数のTSL書き直し＋r169→r171+更新＋iOS初物リスクに対し電力実測の裏付け無し＝**今は見送りが妥当**。
- 電池/熱を読むWeb APIは無し。**フレーム時間の移動平均監視**が唯一の間接検知。

## 4. 効果の概算と検証計画

- **CPU**: A1（▲20ms級）+A2/B1/C1（コール発行▲20〜40%）+B3で、フレームあたりのCPU仕事を**3〜5割削減**の見込み。CPUがヒマになる＝SoC電力・発熱に直結（iPhoneのWebGLはコール発行がCPU仕事）。
- **GPU帯域**: A3（▲50%ポスト）+A4+A5+A6+B4で**3〜4割削減**の見込み。
- **時間軸**: B2で静観時間の総仕事▲20%。
- 数値は開発機実測＋算術見積り。**実機の最終確認はiPhoneをMacのSafariにつなぎWebインスペクタのタイムライン録画**（フレーム内訳のbefore/after）＋「1日プレイして温度の体感」。
- 回帰防止: 各施策1コミット。`_soak`（時間帯別+連続）/`_callsPerSec`/`_matcost`/`_dcattr`/`_dcfog`（後3つは本調査で新設済み・scripts/_*.mjsはgitignore）でbefore/after数値を記録。画素一致はprepass far300と同じA/B法（同設定連続2枚のゆらぎをベースラインに）。CI予算（calls≤4100等）は現状維持でよい（施策はすべて予算を下げる方向）。

### 実装順の推奨（1施策=1コミット・各フェーズ後に_soakと実機確認）

1. **Phase 1（見た目不変・即効）**: A7 → A1 → A2 → A6 → A4
2. **Phase 2（A/B付き・見た目不変）**: A3 → A5 → B4の前段としてB1
3. **Phase 3（ユーザー決定を反映）**: B2 / C2 / C3（§5の回答どおり）
4. **Phase 4（大物・任意）**: B4 → C1（1種ずつ）

### アプリ外の即効Tips（ユーザー向け）

- **iPhoneの低電力モードONで遊ぶ**: SafariのrAFが30fpsになるが、本ゲームは元々30fps上限なので**見た目が一切変わらずOS側の節電だけ効く**。実は今日から使える最強の対策。
- 画面輝度を下げる・ケースを外す（放熱）も物理的に有効。
